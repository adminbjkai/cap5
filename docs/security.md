# Security

This document covers the security model for cap5. The platform is intentionally single-tenant (no end-user authentication), so the threat model focuses on protecting the processing pipeline and object storage rather than user identity.

## HMAC Webhook Signing

Incoming webhook requests (from `media-server` to `POST /api/webhooks/media-server/progress`) are authenticated with HMAC-SHA256 signatures.

### How it works

1. The sender computes `HMAC-SHA256(secret, rawBody)` and includes the result in the `X-Signature-256` header.
2. The receiver re-computes the HMAC using `MEDIA_SERVER_WEBHOOK_SECRET` and the raw request body (captured before JSON parsing via `fastify-raw-body`).
3. The signature is compared with `timingSafeEqual` to prevent timing attacks.
4. The request timestamp (`X-Timestamp` header) is validated against `WEBHOOK_MAX_SKEW_SECONDS` (default: 300s). Requests outside the window are rejected to prevent replay attacks.
5. Each delivery carries a `delivery_id`. Duplicate deliveries are rejected by the `uq_webhook_source_delivery` unique constraint on `webhook_events`.
6. Progress updates within the same `(source, job_id, phase, progress_bucket)` are also deduplicated by `uq_webhook_source_job_phase_bucket`.

### Configuration

| Env var                         | Minimum | Notes                                         |
|---------------------------------|---------|-----------------------------------------------|
| `MEDIA_SERVER_WEBHOOK_SECRET`   | 32 chars| Shared secret for HMAC; validated at startup  |
| `WEBHOOK_MAX_SKEW_SECONDS`      | â€”       | Default 300. Set to 0 to disable skew check   |

The secret is validated by `@cap/config` with `.min(32)` so startup fails fast if too short.

### SSRF protection

The `webhookUrl` field on `videos` (used by the outgoing `deliver_webhook` job) is validated before use:
- Only `http:` and `https:` protocols are allowed
- Hostname blocklist rejects internal service names (e.g., `minio`, `postgres`, `media-server`, `localhost`)

## Rate Limiting

Rate limiting is applied globally via `@fastify/rate-limit`.

Key points:
- The `allowList` callback is registered to allow the rate-limit plugin to work at the plugin level rather than per-route (per-route config is not supported by this plugin version).
- Webhook routes are subject to the same global rate limits.
- Default limits are configured in `packages/config` and can be overridden via environment variables.

For current limit values, see [docs/environment.md](environment.md).

## No-Auth Rationale

cap5 does not implement end-user authentication. This is a deliberate single-tenant design decision:

- The stack is designed to run as a personal or team-internal tool, not a multi-user SaaS.
- Network isolation (Docker Compose, private networking) is the primary access control layer.
- The MinIO console (port `8923`) is bound to `127.0.0.1` only.
- Exposing the API publicly without a reverse proxy and authentication layer in front of it is not recommended for production.

Authentication (JWT/session-based single-user login) is tracked as a future phase and has been explicitly deferred by the project owner.

## S3 Access Patterns

### Internal vs public endpoints

| Env var               | Used by                    | Value in Docker              |
|-----------------------|----------------------------|------------------------------|
| `S3_ENDPOINT`         | Backend (server â†’ MinIO)   | `http://minio:9000` (internal) |
| `S3_PUBLIC_ENDPOINT`  | Presigned PUT URLs / UI    | `http://localhost:8922` (browser-accessible) |

Presigned URLs are generated by the API and handed to the browser. The browser uploads directly to MinIO using the presigned URL â€” the API never proxies the raw bytes.

### Bucket policy

The `minio-setup` service sets the bucket to public-read on first boot. This is intentional for the current single-tenant model: processed video files and thumbnails are served directly from MinIO without requiring signed GET URLs.

The `VITE_S3_PUBLIC_ENDPOINT` build-time variable controls how the frontend constructs playback URLs. In Docker (nginx), leave it unset so the frontend uses relative `/cap5/â€¦` paths (nginx proxies these to MinIO). For Vite dev mode with Docker infra, set it to `http://localhost:8922`.

### What is sandboxed

- MinIO console is localhost-only (`127.0.0.1:8923`)
- PostgreSQL port is configurable (`POSTGRES_PORT`, default `5432`) and should not be exposed publicly
- `media-server` is not accessible to the browser â€” only the `worker` calls it via Docker's internal network
- `web-builder` has CPU and memory limits (`1.0 CPU`, `512 MB`) to prevent resource exhaustion during the build step

### What is not sandboxed

- MinIO API port (`8922`) is exposed on all interfaces by default â€” restrict this in production
- Web API port (`3000`) is exposed on all interfaces â€” put a reverse proxy in front for production
- There is no CSP header configuration in the checked-in nginx config â€” add this for production deployments
